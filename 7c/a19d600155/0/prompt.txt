## What cmux Is

**cmux** is a native macOS terminal application built with Swift and AppKit that extends [Ghostty](https://ghostty.org)'s capabilities. It uses a Ghostty fork (`manaflow-ai/ghostty`) as its rendering engine via `GhosttyKit.xcframework`, meaning it is GPU-accelerated and reads your existing Ghostty config at `~/.config/ghostty/config` (themes, fonts, colors all carry over).

The primary use case is **managing multiple AI coding agents simultaneously** -- think of it as a terminal multiplexer (like tmux) but as a native macOS app with a vertical sidebar, built-in browser, and a rich CLI/socket API designed for agent automation.

Repository: `https://github.com/manaflow-ai/cmux`  
License: AGPL-3.0-or-later  
Latest version: 0.61.0 (2026-02-25)

---

## Core Concepts / Hierarchy

cmux uses a strict hierarchy of objects:

| Concept | Description |
|---|---|
| **Window** | A native macOS window |
| **Workspace** | A sidebar entry within a window (the "tab" in the sidebar) |
| **Pane** | A split region inside a workspace |
| **Surface** | A tab within a pane (the primary automation target) |

Each object has a stable ID. Since v0.53.0, the CLI uses **short ID references** like `surface:1`, `pane:2`, `workspace:3` instead of raw UUIDs.

---

## Key Features

### 1. Vertical Tab Sidebar
The sidebar shows each workspace with:
- Git branch information
- Linked PR status/number
- Working directory
- Active listening ports
- Latest notification text

### 2. Notification System
- Panes get **blue rings** when agents need attention
- Sidebar tabs illuminate for unread notifications
- A dedicated notification panel consolidates all alerts
- `Cmd+Shift+U` jumps to the most recent unread notification

### 3. Integrated Browser
An in-app browser with a scriptable API (ported from `agent-browser`) allows agents to:
- Snapshot accessibility trees
- Obtain element references (`@e1`, `@e2`, etc.)
- Click, fill forms, execute JavaScript
- Navigate, reload, go back/forward

### 4. Session Restoration
On restart, cmux restores window/workspace/pane layouts, working directories, terminal scrollback (best-effort), and browser history/URLs. It does **not** restore live process state (running agents, tmux sessions, vim instances).

### 5. Command Palette
`Cmd+Shift+P` opens a command palette for cross-window tab switching, update actions, and more (added in v0.61.0).

## How to Open/Focus a Specific Tab (Workspace)

This was one of your key questions. There are multiple approaches:

### Keyboard Shortcuts
| Shortcut | Action |
|---|---|
| `Cmd+N` | New workspace |
| `Cmd+1` through `Cmd+8` | Jump to workspace by index |
| `Cmd+9` | Jump to last workspace |
| `Ctrl+Cmd+]` | Next workspace |
| `Ctrl+Cmd+[` | Previous workspace |
| `Cmd+Shift+P` | Command palette (cross-window tab switching) |

### CLI Commands
```bash
# List all workspaces (see IDs and selection status)
cmux list-workspaces

# Select/focus a specific workspace by ID
cmux select-workspace --workspace <id>

# Create a new workspace, optionally running a command in a directory
cmux new-workspace --command "claude --dangerously-skip-permissions" --cwd ~/projects/myrepo

# Rename a workspace
cmux rename-workspace --workspace <id> "My Agent"

# Move a workspace to a different window
cmux move-workspace-to-window --workspace <id> --window <id>
```

### Surface/Pane Focus (within a workspace)
```bash
# Focus a specific pane
cmux focus-pane --pane <id>

# List surfaces in a pane, then focus one
cmux list-pane-surfaces
# Ctrl+1 through Ctrl+9 to jump to surfaces by index

# Focus directionally between split panes
# Option+Cmd+Arrow keys
```

### Environment Variables
cmux automatically sets these in child shells:
- `CMUX_SOCKET_PATH` -- the control socket path
- `CMUX_TAB_ID` -- current tab/workspace UUID
- `CMUX_PANEL_ID` -- current panel/surface UUID
- `CMUX_WORKSPACE_ID` -- used by CLI commands for workspace-relative operations

## Full CLI Command Reference

The CLI binary is `cmux` and communicates with the app via a Unix domain socket.

### Global Flags
| Flag | Description |
|---|---|
| `--socket <path>` | Specify socket location |
| `--json` | JSON output format |
| `--id-format refs\|uuids\|both` | Identifier display mode |
| `--window <id>` | Target specific window |
| `--password <value>` | Socket authentication |
| `-v, --version` | Version info |

### Window Management
```bash

cmux focus-window --window <id>
```


### Pane & Surface Management
```bash

cmux close-surface [--workspace <id>] [--surface <id>]
```

### Splits
```bash

cmux new-pane --direction down
```

### Text Input / Screen Reading
```bash

cmux send-key --surface <id> <key>
cmux read-screen [--surface <id>] [--lines <n>] [--scrollback]
```

### Notifications
```bash

cmux list-notifications

```

### Status & Progress
```bash

```

### Logging
```bash

```

### Browser

## Integration with Terminal / Agent Workflows

### Claude Code Integration

cmux provides a `claude-hook` command for Claude editor hooks with session tracking. You can configure notification hooks in `~/.claude/settings.json`:

```json
{
  "hooks": {
    "matchers": ["idle_prompt", "permission_prompt"],
    "command": "cmux notify --title 'Claude' --body 'Needs attention'"
  }
}
```

A safe fallback pattern for scripts that may run outside cmux:
```bash
command -v cmux &>/dev/null && cmux notify --title "Done" --body "Task complete" \
  || osascript -e 'display notification "Task complete" with title "Done"'
```

### V2 JSON Socket Protocol

For programmatic integration, cmux exposes a JSON-RPC v2 protocol over its Unix socket. Request format:
```json
{"id":"1","method":"workspace.list","params":{}}
```
Response:
```json
{"id":"1","ok":true,"result":{...}}
```

This enables building custom tooling that orchestrates workspaces, sends input, reads screens, manages browser sessions, and handles notifications -- all programmatically.

### Key Design Principle

From the project's own description: "cmux is not prescriptive about how developers hold their tools. It's a terminal and browser with a CLI, and the rest is up to you." It provides composable primitives rather than enforcing specific workflows.
